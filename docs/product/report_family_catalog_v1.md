# Report Family Catalog v1

## Purpose

This catalog names the report families implied by the report-system redesign. It is a design catalog, not an implementation list.

It does not add report views, change advisor UI, generate analytics outputs, wire generated reports, change backend behavior, use real data, or add dependencies.

Each row includes: `report_family_id`, display name, master question family, core question, audience options, likely rendering form, representation level, denominator/category system, data prerequisites, analytics prerequisites, build timing, summary/detail variants, related reports, and notes/caveats.

## Catalog Rules

- A family may have multiple report instances, but each instance should answer one question.
- A report that needs missing data should be marked unavailable, deferred, or advisor-readiness only.
- A table must not combine additive allocations with non-additive warning/status rows.
- Summary/detail variants are separate report choices, not one ever-expanding page.
